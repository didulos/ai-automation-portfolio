  STEP 2: Phase 1 — 병렬 분석 (동시 실행)

  │  ┌────────────┐  ┌────────────────────────┐  │
  │  │doc-updater │  │  automation-scout      │  │

  │  ┌──────────────────┐  ┌──────────────────┐  │
  │  │learning-extractor│  │followup-suggester│  │

  │  └──────────────────┘  └──────────────────┘  │

  ┌─────────────────────────────────┐

## Step 1: 변경 내용 확인

다음 두 명령어를 실행하세요:

```bash
git status
git diff --stat HEAD~3 2>/dev/null || git diff --stat
```

결과를 바탕으로 아래 항목을 파악하세요:
- **변경된 파일**: 수정된 파일 목록
- **새로 생성된 파일**: 추가된 파일 목록
- **삭제된 파일**: 제거된 파일 목록

변경 내용을 한 줄로 요약한 뒤 아래 Session Summary를 작성하세요.

---

## Session Summary 작성

git status/diff 결과를 바탕으로 템플릿을 채운다. **이 요약을 Phase 1의 4개 에이전트 모두에게 전달한다.**

```
Session Summary:
- Work: [오늘 수행한 주요 작업 2-3줄]
- Files: [생성/수정된 파일 목록]
- Decisions: [내린 주요 결정]
```

---

## Phase 1: 병렬 분석

아래 4개 에이전트를 **한 메시지에서 4개의 Task 호출로 동시에(병렬로)** 실행하세요. 위에서 작성한 Session Summary를 각 에이전트 프롬프트에 함께 전달하세요. 각 결과를 별도로 저장하세요.

**에이전트 1 — doc-updater**
> CLAUDE.md, README 등 문서 파일을 읽고, 오늘 변경된 내용을 기준으로 업데이트가 필요한 항목을 탐색한다. 업데이트 대상 파일명과 수정이 필요한 내용을 목록으로 반환한다.

**에이전트 2 — automation-scout**
> 오늘 작업 내역을 분석해 반복된 패턴을 찾는다. 자동화 스킬 또는 셸 스크립트로 만들 수 있는 후보 작업을 제안한다. 후보 이름과 이유를 목록으로 반환한다.

**에이전트 3 — learning-extractor**
> 오늘 세션에서 새로 배운 것, 새로 알게 된 패턴, 시행착오를 정리한다. 학습 항목 목록을 반환한다.

**에이전트 4 — followup-suggester**
> 미완성 작업, 연결된 다음 작업, 다음 세션에서 이어갈 항목을 제안한다. 우선순위 순으로 목록을 반환한다.

---

## Phase 2: 검증

**에이전트 — duplicate-checker**

Phase 1 결과를 아래 형식으로 전달하고 순차적으로 검증하세요:

```
Phase 1 결과:

## doc-updater 제안:
[doc-updater 결과]

## automation-scout 제안:
[automation-scout 결과]

## learning-extractor 결과:
[learning-extractor 결과]

## followup-suggester 제안:
[followup-suggester 결과]
```

검증 기준:
- **완전 중복** → 하나를 제거 (이미 CLAUDE.md에 있는 내용, 기존 스킬과 동일한 자동화)
- **부분 중복** → 합쳐서 하나로 병합
- **고유 항목** → 그대로 유지

중복 제거 후 최종 실행 가능 항목 목록을 작성하세요.

Phase 1 + Phase 2 결과를 아래 **정확한 포맷**으로 출력하세요.

📄 문서 업데이트 필요:
  - [항목들]

⚙️  자동화 제안:
  - [항목들]

🎓 오늘 배운 것:

## 커밋 생성 (선택된 경우 실행)

사용자가 "커밋 생성"을 선택했을 때 아래 순서로 실행하세요.

```bash
# 1. 변경 파일 확인
git status

# 2. 변경 내용 스테이징 (새 파일 포함)
git add [변경된 파일들]  # -A 대신 파일명 명시 권장

# 3. 커밋 메시지 형식 (한국어, Conventional Commits 스타일)
# feat:  새 기능 추가
# fix:   버그 수정
# docs:  문서 변경
# chore: 설정/빌드 변경
git commit -m "$(cat <<'EOF'
[타입]: [오늘 주요 변경 한 줄 요약]

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
EOF
)"

# 4. 커밋 확인
git log --oneline -3
```

**커밋 메시지 예시:**
- `feat: my-history-insight 스킬 + history_insight.py 추가`
- `docs: CLAUDE.md Repository Structure + Scripts 섹션 업데이트`
- `chore: my-session-wrap SKILL.md 커밋 명세 보강`

## 사용 방법 (Arguments)

| 호출 방식 | 동작 |
|----------|------|
| `/wrap` | 전체 대화형 워크플로우 (분석 → 선택 → 실행) |
| `/wrap "feat: ..."` | 분석 없이 바로 커밋 메시지로 커밋 생성 |

**빠른 커밋 예시:**
```
/wrap "feat: my-fetch-youtube 임시파일 경로 + 언어 감지 분기 개선"
```
→ Phase 1/2 분석 생략, 바로 git add + commit 실행

---

## 언제 사용할까

**사용 권장:**
- 의미 있는 작업 세션이 끝날 때
- 다른 프로젝트로 전환하기 전
- 기능 완성 또는 버그 수정 완료 후

**생략해도 되는 경우:**
- 코드 읽기/탐색만 한 짧은 세션
- 변경사항이 없는 경우
- 간단한 일회성 질문만 한 경우
